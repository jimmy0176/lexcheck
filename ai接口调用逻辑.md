请改造“快速体检报告”的附件处理逻辑，目标是避免当前对长文档做 22k 字符硬截断，改为“分块提取 + 汇总生成”的方式。

一、当前问题
现在 quick-exam-report 的逻辑是：
1. 读取 prompt.md
2. 读取 output.md
3. 读取三方速查附件正文
4. 读取问卷 narrative
5. 拼接后直接提交给大模型

其中三方速查附件正文当前是多文件用 --- 拼接，并在总长约 22k 字符处截断。
这个实现会导致：
- 超长附件后半部分内容被直接丢弃
- 多文件时靠后的文件可能完全未进入模型
- 法律风险、经营风险、资质、诉讼等后半部分信息容易遗漏

二、改造目标
不要再对附件正文做简单硬截断。
请改为两阶段流程：

阶段1：附件预处理摘要
- 按文件分别读取附件正文
- 若单个文件过长，则按固定长度切块
- 对每个文件或每个分块，调用大模型提取结构化摘要
- 摘要只提取关键信息，不直接生成最终报告

阶段2：最终报告生成
- 将所有附件摘要结果合并
- 再结合 prompt.md、output.md、问卷 narrative
- 统一交给大模型生成“快速体检报告”

三、实现要求

1. 附件分块逻辑
请新增一个通用分块函数，例如：
splitTextIntoChunks(text, maxChars)

要求：
- 不按 22k 直接截断全文
- 单块建议控制在 6000~8000 中文字符以内
- 尽量优先按自然边界切分，例如：
  - 标题
  - 空行
  - 段落
- 如果找不到合适边界，再按字符长度切分
- 返回 chunk 列表

2. 单块摘要提取
请为每个 chunk 调用大模型，输出严格 JSON，不要输出 Markdown。

建议 JSON 结构如下：

{
  "sourceFileName": "",
  "chunkIndex": 0,
  "basicInfo": [],
  "shareholders": [],
  "management": [],
  "investments": [],
  "changeRecords": [],
  "legalRisks": [],
  "businessRisks": [],
  "qualifications": [],
  "licenses": [],
  "taxInfo": [],
  "employmentInfo": [],
  "ipInfo": [],
  "otherImportantFindings": []
}

提取规则：
- 仅提取对快速体检报告有价值的信息
- 即使内容是“0”“无”“未发生”，如果对风险判断、合规状态、企业现状有意义，也要保留
- 不要编造
- 缺失写“未提供”
- 保留来源文件名和 chunkIndex，方便追溯

3. 摘要合并
请新增一个 mergePreliminarySummaries 函数，把多个 chunk 的摘要结果合并为一个总摘要对象。

要求：
- 同类字段合并到一起
- 尽量去重
- 保留来源信息
- 不需要在这一步生成自然语言报告

4. 最终生成
最终调大模型时，输入应改为：
1. prompt.md
2. output.md
3. 问卷 narrative
4. 附件汇总摘要（不是附件全文）

不要再直接传全文附件正文。

5. 前端展示和日志
请在前端或调试日志中增加以下信息：
- 本次附件总文件数
- 每个文件是否分块
- 分块数量
- 是否发生超长分块处理
- 最终参与“最终报告生成”的是“附件摘要”，不是“附件全文”

6. 导出功能保持兼容
快速体检报告右上角下载按钮导出 md / word / pdf 的逻辑保持不变。
本次改造只调整“生成报告前的附件处理逻辑”。

四、代码组织建议
请优先考虑新增或改造以下模块（命名可调整，但职责要清晰）：

- web/lib/quick-exam-preliminary.ts
  负责：
  - 读取附件正文

  负责：
  - 读取 prompt/output/questionnaire

  - 最终生成快速体检报告

五、验收标准
改造完成后，应满足以下效果：

1. 上传一份较长的企查查专业版报告时，不会再因为 22k 截断而丢掉后半部分内容
2. 上传多份附件时，靠后的文件也能进入预处理摘要流程

   - 摘要合并后的结果规模

- 快速体检报告生成入口

## 七、混合策略阈值与异步任务（实现约定）

以下为与评估结论一致的工程约定，便于前后端与验收对齐。

### 7.1 混合策略（同步全文 vs 异步分块）

| 参数 | 含义 | 建议默认值（可通过环境变量覆盖） |
|------|------|----------------------------------|
| `QUICK_EXAM_SYNC_MAX_TOTAL_CHARS` | 三方速查（preliminary）附件正文**合计字符数**上限，低于等于则走**同步单阶段**（一次请求内完成，附件以全文传入终稿模型，**不做 22k 硬截断**） | `48000` |
| `QUICK_EXAM_SYNC_MAX_FILES` | 同步路径允许的 preliminary **文件个数**上限 | `12` |

**判定**：当 `文件数 ≤ SYNC_MAX_FILES` 且 `合计正文长度 ≤ SYNC_MAX_TOTAL_CHARS` 时，走同步路径；否则创建异步任务，按「分块摘要 → 合并 → 终稿」执行。

**说明**：同步路径仅去掉旧的 `slice(0, 22000)`；若单请求仍超出上游模型或网关限制，由异步路径承接。

### 7.2 分块参数

| 参数 | 含义 | 建议值 |
|------|------|--------|
| `QUICK_EXAM_CHUNK_TARGET_CHARS` | 单块目标长度（中文为主） | `7000`（约 6000–8000 区间） |

分块实现须优先按标题、空行、段落等自然边界切分，必要时再按字符硬切。

### 7.3 异步任务状态（`QuickExamReportJob` / `progressJson`）

**任务级字段（数据库）**：

- `id`：任务 ID，客户端凭此轮询/续跑。
- `checkupId`：关联体检单。
- `status`：`pending` | `running` | `success` | `failed`。
- `mode`：`sync_full` | `async_chunk`（仅作记录/展示）。
- `reportText`：成功时写入终稿 Markdown。
- `errorMessage`：失败时写入可读错误。
- `progressJson`：见下。

**progressJson 建议结构**（实现可扩展，前端展示「文件数、分块数、是否分块管线、当前进度」）：

- `fileCount`：preliminary 文件数。
- `totalChunks`：全部分块总数。
- `chunkCursor`：已完成摘要的分块游标（0..totalChunks）。
- `usedChunkPipeline`：是否走分块摘要管线（异步路径为 `true`）。
- `files`：`{ fileName, chunkCount, chunked }[]`。
- `stage`：`chunks` | `final` | `done`。
- `mergedSummaryChars`（可选）：合并后摘要 JSON 字符串长度，用于观察规模。

**安全**：不在服务端持久化用户 API Key；续跑异步步骤时由客户端在每次请求中携带与现有一致的大模型参数。

### 7.4 异步续跑方式

客户端在创建异步任务后，使用**同一 POST 端点**携带 `jobId` 与原有 `promptMd` / `outputMd` / 大模型配置**分步推进**任务，直至返回 `done: true` 与 `reportText`（见实现中的响应约定）。

### 7.5 阶段 1 JSON 容错

对每块结构化摘要要求：解析失败时**自动重试**（有限次数）；在供应商支持时可选 `response_format: json_object`；仍失败时对模型输出做**提取首个 JSON 对象**的修复尝试。详见 `web/lib/quick-exam-json.ts` 实现。
